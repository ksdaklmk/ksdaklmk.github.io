Mirai API Specifications - Alternative Channels for KYC

version 2.0 10-October-2019

This Mirai API specifications document is to support customer identification on the selected alternative channels for the purpose of KYC.

1. Application architecture and sequences diagram

2. Generate QR transaction reference API

This service is to request DIIS to generate QR and barcode transaction reference for the requesting channel.

POST /native/ndid/alternativeSource/initial

Development: https://sit.api.krungsri.net/native/ndid/alternativeSource/initial
(API key: l726df48afe64548c9acb2430af3484009)

Sandbox: https://sandbox.api.krungsri.net/native/ndid/alternativeSource/initial
(API key: l71ad26790bd0f4b9d81b215670b54fb3e)

Request header

Name Type Required Description
Content-Type String Yes application/json
X-Client-Transaction-ID String Yes UUID generated by the client
X-Client-Transaction-DateTime String Yes Date and time on the client side in ISO8601 format
API-Key String Yes Assigned API key for client authentication

Request body

Name Type Required Description
productCode String (10) Yes "OPENACC" for opening saving account
"OPENLOAN" for opening loan account
"OPENCREDIT" for opening credit card account
transactionRef String (18) Yes Channel transaction reference number
identifier String (13) Yes Citizen ID 13 digits
identifierType String (10) Yes "citizen_id"
qrValidity Number (5) Yes Validity of QR in minutes
Maximum 10080 minutes (7 days)
additionalData String (20) No Additional data reserved for future use

Response

Name Type Description
statusCode String (4) status code
statusMessage String Status message
data Object data object
data.qrReference String (128) QR data
data.barcodeReference String (128) Barcode data
data.uuidReference String (20) UUID reference number

Sample request

curl -X POST \
  https://sandbox.api.krungsri.net/native/ndid/alternativeSource/initial \
  -H 'API-Key: l7333hd...' \
  -H 'Content-Type: application/json; charset=UTF-8' \
  -H 'X-Client-Transaction-ID: a01fa0ae-190a-49d5-9cc1-465f9f9ab0dc' \
  -H 'X-Client-Transaction-DateTime: 2019-10-04T12:00:00.000+07:00' \
  -d '{
  "productCode": "OPENACC",
  "transactionRef": "BAYMK190800098",
  "identifier": "3839900145640",
  "identifierType": "citizen_id",
  "qrValidity": 1440,
  "aditionalData": ""
}'

Sample response

{
  "statusCode": "0000",
  "statusMessage": "Successful",
  "data": {
    "qrReference": "XXXXXXXXXXXX",
    "barcodeReference": "XXXXXXXXXXXX",
    "uuidReference": "XXXXXXXXXXXX"
  }
}

5. Error message and error codes

Sample error response

{
  "error": {
    "code": "100",
    "message" : "Description and detail of error",
    "messageTH" : "ข้อความที่ต้องการให้แสดงต่อผู้ใช้งานเป็นภาษาไทย",
    "serverDateTime": "2019-09-01T18:39:35.45+07:00",
    "clientTransactionID": "UUID submitted by the Client",
    "serverTransactionID": "UUID generated by the Server"
  }
}

Note that serverDateTime, clientTransactionID and serverTransactionID are equal to the respective HTTP headers X-Server-DateTime, X-Client-Transaction-ID and X-Server-Transaction-ID respectively.

The code value and its associated description gives a clear indication of the problem. The table below describes the standard possible problems and how to address them. It has been designed with the minimum HTTP status to handle.

HTTP status code Reason phrase Details
200 OK Standard response for successful HTTP requests
400 Bad Request - Invalid request structure (the request cannot be parsed)
- Invalid parameter in the request (i.e. account number cannot be found)
- Submitted data does not match the specification, it can also be considered as a possible attack
401 Unauthorized Authentication failed
500 Internal Server Error Transaction failed. Something went wrong technically in the gateway or in the back-end, or simply some business logic prevents execution (i.e. fund insufficient).
503 Service Unavailable Krungsri server side is currently saturated
